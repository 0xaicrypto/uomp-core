# UOMP Remote Authorization Design Document

> Status: Phase 1 implemented (uomp-mvp `apps/gateway`)
> Objective: Define the secure authorization, communication, and payload delivery mechanism between a user's local Memory and a remote Agent.

---

## 1. Background and Objectives

UOMP's standard model assumes the Agent runs locally on the user's machine or on a trusted machine in the same environment. In real-world scenarios, the Agent is more likely to be:

- A third-party cloud service;
- An open-source community-hosted SaaS;
- A service deployed by the user on another machine or container.

This requires solving three core problems:

1. **Communication Security**: How does the user communicate with a remote Agent while ensuring the Manifest, Token, and Payload are not leaked?
2. **Authorization Mechanism**: How can a locally issued Capability Token be securely handed to a remote Agent without directly exposing the local Memory Guard to the public internet?
3. **Payload Storage**: How can analysis reports generated by the Agent be returned to the user without remaining on third-party servers for an extended period?

This document proposes a unified solution based on **Remote Profile + UOMP Gateway**.

---

## 2. Threat Model

| Threat | Description | Protection Goal |
|---|---|---|
| Man-in-the-middle eavesdropping on Manifest/Token | Attacker monitors the link between user and Agent | TLS 1.3 / DIDComm end-to-end encryption |
| Agent identity forgery | Malicious Agent impersonates a trusted Agent | DID / GPG / X.509 identity verification + Registry validation |
| Token replay after interception | Token is leaked during transit or on the Agent side | Token bound to Gateway, short validity, Gateway-layer revocation |
| Remote Agent directly accessing local Memory Guard | User's internal network is penetrated | Memory Guard is not directly exposed; all access goes through Gateway |
| Payload retained long-term by third party | Report remains on Agent server | E2E encryption + temporary Relay + on-chain hash verification |
| User denies authorization or Agent denies access | No evidence available after the fact | Audit logs + on-chain event anchoring |

Key principles:

- **Memory Guard is not directly exposed to the public internet**; the Gateway is the only external entry point.
- **The audience of the Capability Token is the Gateway**, not the Memory Guard.
- **Payloads must be end-to-end encrypted**; the Relay stores only ciphertext.

## 4. Communication Mechanism

### 4.1 Agent Discovery and Manifest Retrieval

The user discovers a remote Agent via the Registry or a URL:

```bash
uomp discover registry://stock-analyst
# or
uomp discover https://agent.example.com/uom.json
```

Security requirements:

1. **TLS 1.3 mandatory**: All Manifest downloads must use HTTPS.
2. **Identity verification**: The Agent must provide a verifiable identity (DID / GPG / X.509).
3. **Integrity verification**: Compute the checksum of `uom.json` and compare it against Registry or on-chain records.
4. **Do not send user data**: The discovery phase only transmits Agent ID / URL; no Memory data is uploaded.

### 4.2 Secure Channel Selection

| Option | Applicable Scenario | Pros and Cons |
|---|---|---|
| **mTLS HTTPS** | Default recommendation | Mature and easy to deploy; requires certificate distribution and revocation mechanism |
| **DIDComm v2** | High-security / decentralized scenarios | End-to-end encryption, no pre-shared keys needed; ecosystem is relatively new |
| **Noise + WebSocket** | Real-time bidirectional streaming | Lightweight; requires custom protocol |

**MVP recommendation: mTLS HTTPS**:

- The Agent registers its TLS client certificate or certificate fingerprint in the Registry.
- When the user runs `uomp connect`, the client verifies that the Agent certificate matches the Registry record.
- Subsequent authorization negotiation and Token delivery take place within this mutually authenticated TLS channel.

### 4.4 DIDComm Migration Roadmap

In the long term, UOMP plans to migrate from mTLS to **DIDComm v2**, because:

- True end-to-end encryption without relying on TLS certificate authorities;
- Agent and user do not need to pre-share keys;
- Supports asynchronous messages, offline authorization, and multi-device synchronization.

Migration strategy (backward compatible):

1. **Phase 1 (MVP)**: mTLS HTTPS is the default channel; DIDComm is an optional extension.
2. **Phase 2**: Agents publish both an mTLS endpoint and a DIDComm service endpoint; the client prefers DIDComm.
3. **Phase 3**: New Agents may support only DIDComm; legacy Agents continue to be supported through an adapter layer for mTLS.

Reserved DIDComm message types:

- `uomp/authorize-request`
- `uomp/authorize-response`
- `uomp/payload-ready`
- `uomp/session-revoked`

Deployment options:

- A reverse tunnel (Cloudflare Tunnel, ngrok) can also be used to expose the Gateway to the public internet without requiring a public IP.
- Communication between Gateway and local Memory Guard uses mTLS or a local Unix socket.
- **Future commercial option**: A managed Gateway service may be offered, but the open-source core will always retain self-hosting as the default and trust root.

## 5. Authorization Mechanism

### 5.1 Gateway-Bound Capability Token

When the user performs authorization, the Token's `audience` points to the Gateway rather than the local `127.0.0.1`:

```ts

  profile: 'remote',

    '/v1/memory/*',
```

### 5.3 Gateway Validation and Forwarding

For each request, the Gateway performs the following:

1. Validate whether the TLS client certificate is on the allowlist;
2. Validate the signature, expiration, and revocation status of `Authorization: Bearer <token>`;
3. Validate whether the request path is within `allowedEndpoints`;
4. Validate whether the `limits` quota is exceeded;
5. Forward the request to the local Memory Guard;
6. Record an audit log;
7. Optionally anchor the access event on-chain.

### 5.4 Revocation

After the user runs `uomp revoke sess_xxx`:

- The local Auth Service marks the session as `revoked`;
- The Gateway periodically syncs the denylist, or the Auth Service pushes revocation events to the Gateway;
- The Gateway immediately rejects subsequent requests using that Token.

### 5.5 Token Refresh Mechanism

Remote Agents may run for an extended period (for example, background analysis tasks), so short-lived Tokens need to be refreshed:

1. At initial authorization, the Auth Service issues both an `access_token` and a **refresh_token**:
   - The refresh_token is only used to obtain a new `access_token`;
   - Its scope is fixed and it cannot be used to read Memory;
   - Its validity period is configurable (default 7 days).
2. Before the `access_token` expires, the Agent calls the Gateway refresh endpoint:
   ```http
   POST /v1/sessions/{session_id}/refresh
   Authorization: Bearer <refresh_token>
   ```
3. After verifying the refresh_token and session status, the Gateway requests a new `access_token` from the local Auth Service and returns it.
4. When the user revokes the session, the refresh_token becomes invalid simultaneously.

> The refresh mechanism allows the remote Agent to avoid holding the same JWT for a long time, while keeping Token lifecycle control on the user side.

## 6. Payload Storage and Delivery

### 6.1 End-to-End Encryption

- The user provides a public key in the Remote Profile.
- After generating a report, the Agent encrypts it using that public key (for example, ECDH-X25519 + AES-GCM-256).
- Only the user's local private key can decrypt it; the Relay / Gateway stores only ciphertext.

**CLI Private Key Protection**:

- The decryption private key is stored by default under `~/.uomp/.keystore/` as an encrypted keyfile (encrypted with a password-derived key).
- The key pair is generated when `uomp init` is first run, and the user is prompted to set a keystore password.
- When decrypting a Payload, the CLI temporarily unlocks the private key and clears it from memory immediately after the operation.
- GUI applications may use the system keychain; the CLI uses keystore file + password.

### 6.2 Payload Relay Options

| Mode | Flow | Applicable Scenario |
|---|---|---|
| **gateway-cache** | Agent POSTs encrypted Payload to Gateway; user pulls from Gateway | Synchronous, small files (< 10 MB) |
| **presigned-url** | Gateway gives Agent a one-time upload URL; Agent uploads and returns reference | Large files, asynchronous tasks |
| **ipfs-cid** | Ciphertext uploaded to IPFS; CID + hash stored on-chain | Publicly verifiable, long-term archive |

Recommended default: **gateway-cache** + on-chain hash anchoring.

### 6.3 Payload Envelope

```json
{
  "payload_id": "pay_xxx",
  "session_id": "sess_xxx",
  "agent_id": "stock-analyst",
  "timestamp": "2026-07-14T10:15:00Z",
  "encryption": {
    "algorithm": "ECDH-X25519-AES256GCM",
    "sender_public_key": "...",
    "nonce": "..."
  },
  "ciphertext": "base64...",
  "format": "text/markdown",
  "size": 2048,
  "hash": "sha256:..."
}
```

After receiving the Payload, the user:

1. Decrypts it with the local private key;
2. Verifies that `hash` matches the content;
3. Optionally verifies the on-chain anchored hash.

## 7. Auditing and On-Chain Anchoring

### 7.1 Local Auditing

The Gateway writes the following events to the local `audit_remote.db`:

| Event | Fields |
|---|---|
| `gateway_access` | session_id, agent_id, endpoint, allowed, reason, timestamp |
| `payload_upload` | payload_id, session_id, agent_id, hash, timestamp |
| `payload_download` | payload_id, timestamp |

### 7.2 On-Chain Anchoring

For tamper-proof evidence, key events are written to blockchain events:

```solidity
event SessionGranted(bytes32 indexed sessionHash, bytes32 indexed agentHash, uint256 expiresAt);
event GatewayAccess(bytes32 indexed sessionHash, bytes32 indexed agentHash, bytes32 endpointHash, bool allowed);
event PayloadAnchored(bytes32 indexed payloadHash, bytes32 indexed sessionHash);
```

- `sessionHash` = `keccak256(session_id + agent_id + granted_tags)`
- `agentHash` = `keccak256(agent_id)`
- `payloadHash` = `sha256(ciphertext)`

Default support for **Starknet** (low cost, high-frequency events), with optional EVM-compatible chains.

**Batch On-Chain Strategy**:

- **Authorization events** (`SessionGranted`, `SessionRevoked`): Recommended to be anchored in near real-time because they are low-volume and high-criticality.
- **Access events** (`GatewayAccess`): High-frequency, so they are batched. Every N minutes or every M events, the Gateway generates a Merkle root and anchors the root on-chain; the original logs remain on the Gateway for later verification.
- **Payload anchoring**: Each Payload's hash is anchored individually after generation, because the volume is relatively low.

```

                          в”Ӯ
                          в””в”Җв”Җ Full logs retained on Gateway for audit verification
```

## 8. Interface Draft

### 8.1 Gateway HTTP API (Phase 1 implemented)

```http
GET  /v1/health
GET  /v1/memory/{key}              # Forward to Memory Guard
GET  /v1/memory?tag={tag}          # Forward to Memory Guard
GET  /v1/audit                     # Forward to Memory Guard audit endpoint

GET  /v1/payload/{payload_id}      # User downloads Payload
POST /v1/sessions/{session_id}/refresh  # Token refresh (reserved, not implemented)
```

All requests must carry:

```http
Authorization: Bearer <gateway-token>
X-UOMP-Agent-Id: stock-analyst
```

```ts
const report = await agent.analyze(holdings);

// Upload Payload after encryption
await agent.output.upload(report, { encryptTo: remoteProfile.encryption.public_key });
```

## 10. Confirmed Decisions

1. **Gateway Deployment**: The open-source core defaults to user self-hosting; a managed Gateway service may be offered commercially in the future.
2. **Token Refresh**: Introduce refresh_token so the Agent can request a new Token after the access_token expires.
3. **CLI Private Key Protection**: Use keystore file + password-derived encryption; GUI uses system keychain.
4. **Channel Migration**: MVP defaults to mTLS HTTPS; long-term migration to DIDComm v2 with a compatibility layer retained.
5. **On-Chain Anchoring**: Authorization events are anchored near real-time; access events are anchored via batch Merkle root; Payload hashes are anchored individually.
